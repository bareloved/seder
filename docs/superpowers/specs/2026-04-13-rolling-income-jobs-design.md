# Rolling Income Jobs — Design

**Status:** Approved, ready for implementation plan
**Date:** 2026-04-13
**Author:** Brainstorming session w/ Barel
**Scope:** Web (Next.js) + iOS (SwiftUI) + `@seder/shared`

## Problem

Seder users (freelancers, musicians) have recurring income sources that today require manually typing a new entry every time: weekly private students, monthly club gigs, daily delivery runs. There is no concept of a recurring template. Users who happen to have the recurrence in Google Calendar get one draft per instance via calendar import, but (a) not every user connects Calendar, (b) not every recurring income is a calendar event, and (c) editing the rate going forward means editing every future draft by hand.

## Goal

Let a user define a **rolling job** once — title, cadence, amount, client, category — and have Seder auto-materialize future income entries on a rolling horizon. Editing the template should propagate to still-attached future entries. Per-row edits should silently detach so the user's manual adjustments never get clobbered.

## Non-goals (explicitly out of v1)

- Monthly-by-weekday ("first Saturday of every month")
- Yearly cadence
- Multiple amounts within a single cadence ("₪200 regular weeks, ₪100 off weeks")
- Two-way calendar sync (editing an income row writing back to Google Calendar)
- Retroactive materialization of a paused period on resume

## Key design decisions

1. **Pre-generated real rows.** Each occurrence is a real `income_entries` row. Every existing feature (KPIs, analytics, nudges, client pages, exports, calendar dedup) keeps working without special-casing virtual rows.
2. **Rolling horizon, cron-maintained.** A nightly cron tops up each active rolling job so there are always ~3 months of future drafts materialized. Optional explicit end date stops generation earlier.
3. **Per-row edits silently detach.** A `detachedFromTemplate` flag on `income_entries` flips to `true` the first time a user edits a template-tracked field on that row. Once detached, future template edits never touch that row. Matches the Google/Apple Calendar mental model, zero modal dialogs on normal edits.
4. **Calendar promotion is first-class.** Recurring Google Calendar events can be promoted to rolling jobs. The rolling job stores the Google `recurringEventId`, and the calendar importer skips any event whose parent recurring ID is already owned by a rolling job for that user.
5. **Cadence: daily / weekly-with-weekdays / monthly-by-day-of-month.** Covers the real cases ("weekdays only" = weekly + Sun–Thu in Israel) without full RRULE complexity. Tagged union stored as JSONB, validated by Zod in `@seder/shared`.

## Data model

### New table: `rolling_jobs`

```ts
{
  id: uuid pk
  userId: text fk user
  isActive: boolean default true                 // pause = false; cron stops topping up
  title: text                                    // internal label, e.g. "דן — שיעור פסנתר"
  description: text                              // copied to income_entries.description
  clientId: uuid fk clients nullable             // optional link to clients table
  clientName: text                               // snapshot (mirrors income_entries pattern)
  categoryId: uuid fk categories nullable
  amountGross: numeric(12,2)
  vatRate: numeric(5,2) default 18
  includesVat: boolean default true
  defaultInvoiceStatus: varchar(20) default "draft"  // reserved for future flexibility
  cadence: jsonb                                 // tagged union, see below
  startDate: date
  endDate: date nullable                         // optional hard stop
  sourceCalendarRecurringEventId: text nullable  // set iff promoted from Google Calendar
  sourceCalendarId: text nullable
  notes: text nullable
  createdAt: timestamp
  updatedAt: timestamp
}
```

Indexes:
- `rolling_jobs (userId, isActive)`
- `rolling_jobs (userId, sourceCalendarRecurringEventId)` — fast importer dedup lookup

### New columns on `income_entries`

- `rollingJobId uuid fk rolling_jobs nullable` — set on every row generated by a rolling job.
- `detachedFromTemplate boolean default false` — flips to `true` the first time a user edits `description`, `amountGross`, `vatRate`, `includesVat`, `date`, `clientId`, or `categoryId` on that row. Once `true`, template edits never touch this row again.

New partial unique index: `income_entries (rollingJobId, date) WHERE rollingJobId IS NOT NULL` — serves double duty as the lookup index for "find still-attached future rows to rewrite" and as the uniqueness guarantee that makes `INSERT ... ON CONFLICT DO NOTHING` idempotent.

### Cadence type (in `@seder/shared`)

```ts
export type Cadence =
  | { kind: "daily";   interval: number }                           // every N days
  | { kind: "weekly";  interval: number; weekdays: number[] }       // 0=Sun..6=Sat
  | { kind: "monthly"; interval: number; dayOfMonth: number };      // 1..31, clamped
```

Zod schema mirrors this and lives in `packages/shared/src/schemas/rollingJob.ts`. The schema module also exports the full `RollingJob` DTO consumed by the API client, the web app, and the iOS Swift Codable models.

## Architecture

### Pure generator — `@seder/shared`

```ts
// packages/shared/src/rolling-jobs/generate.ts
export function generateOccurrences(input: {
  cadence: Cadence;
  startDate: Date;       // inclusive
  endDate?: Date;        // exclusive, from rolling_jobs.endDate
  horizonEnd: Date;      // today + 3 months, set by caller
  skipBefore?: Date;     // exclude occurrences already materialized
}): Date[];
```

- Pure function, no DB, no side effects.
- Walks forward from `startDate` in cadence-sized steps.
- `weekly.weekdays`: emits all checked weekdays in each `interval`-week block that fall in range.
- `monthly.dayOfMonth`: clamps to last day of month when the target day doesn't exist (Feb 30 → Feb 28/29).
- Hard cap of 400 returned occurrences per call — protection against pathological configurations.
- All dates interpreted in the user's `userSettings.timezone` (default `Asia/Jerusalem`). We store `date` columns (not timestamps) so DST doesn't shift anything.

### Server-side materialization — `apps/web`

```ts
// apps/web/lib/rollingJobs/materialize.ts
export async function materializeRollingJob(
  job: RollingJob,
  opts: { horizonEnd: Date }
): Promise<number>  // rows inserted
```

1. `SELECT MAX(date) FROM income_entries WHERE rollingJobId = job.id` → `lastMaterialized`.
2. Call `generateOccurrences` with `skipBefore = lastMaterialized + 1 day` and `horizonEnd = min(today + 90d, job.endDate ?? +∞)`.
3. Bulk insert in a single transaction using `INSERT ... ON CONFLICT (rollingJobId, date) DO NOTHING`.
4. Each new row defaults:
   - `invoiceStatus = "draft"`
   - `paymentStatus = "unpaid"`
   - `amountPaid = "0"`
   - `detachedFromTemplate = false`
   - `description`, `clientId`, `clientName`, `categoryId`, `amountGross`, `vatRate`, `includesVat`, `notes` copied from the template
   - `rollingJobId = job.id`
5. Returns the row count. Idempotent by design.

### Nightly cron — `app/api/cron/rolling-jobs/route.ts`

- Runs once per day. Pattern matches existing `overdue-notifications` and `backup` crons.
- Guarded by `CRON_SECRET` header check (same helper used by the other crons).
- Pseudocode:
  ```ts
  const jobs = await db.select().from(rollingJobs).where(eq(rollingJobs.isActive, true));
  let processed = 0, inserted = 0, failed = 0;
  for (const job of jobs) {
    try {
      const n = await materializeRollingJob(job, { horizonEnd: addDays(today, 90) });
      processed++;
      inserted += n;
    } catch (err) {
      failed++;
      Sentry.captureException(err, { tags: { cron: "rolling-jobs", jobId: job.id } });
    }
  }
  log({ processed, inserted, failed });
  ```
- One failed job does not abort the run.

### Server actions / API routes

New in `apps/web/app/rolling-jobs/actions.ts` (and mirrored under `app/api/v1/rolling-jobs/` for iOS):

- `createRollingJobAction(input)` — validates, inserts the template, runs `materializeRollingJob` synchronously so the user sees the generated rows in the same request, returns `{ job, insertedCount }`.
- `updateRollingJobAction(id, input)` — diffs old vs new:
  - **Cadence / startDate / endDate changed** — recomputes expected future occurrences (from `today` forward) and reconciles attached rows:
    - Deletes attached future rows whose `date` is no longer in the expected set and `detachedFromTemplate = false`.
    - Inserts attached future rows for new expected dates.
    - *Today's row and all past rows are never touched.*
  - **Field-only change** (`amountGross`, `vatRate`, `includesVat`, `description`, `clientId`, `categoryId`, `notes`) — single `UPDATE` on `income_entries WHERE rollingJobId = ? AND date >= today AND detachedFromTemplate = false`.
  - UI wraps the action in a confirmation dialog: "עדכון ישפיע על N רשומות עתידיות. להמשיך?"
- `deleteRollingJobAction(id, opts)` — `opts.deleteFutureDrafts: boolean`.
  - **Keep past (default):** `DELETE FROM rolling_jobs WHERE id = ?`; `UPDATE income_entries SET rollingJobId = NULL, detachedFromTemplate = true WHERE rollingJobId = ?`. The FK on `income_entries.rollingJobId` must be `ON DELETE SET NULL`.
  - **Delete future drafts:** same as above, but first `DELETE FROM income_entries WHERE rollingJobId = ? AND date > today AND paymentStatus = 'unpaid' AND invoiceStatus = 'draft'`.
- `pauseRollingJobAction(id)` / `resumeRollingJobAction(id)` — just flip `isActive`. No row changes.

`updateIncomeEntryAction` (existing) gets one extra line: when a row with `rollingJobId != null` is updated and a template-tracked field changes, the same `UPDATE` also sets `detachedFromTemplate = true`.

### Calendar integration

#### Importer dedup (during a calendar sync)

The existing importer builds an event list from Google Calendar events. Before inserting each event:

```ts
if (event.recurringEventId) {
  const owned = await db.select({ id: rollingJobs.id })
    .from(rollingJobs)
    .where(and(
      eq(rollingJobs.userId, userId),
      eq(rollingJobs.sourceCalendarRecurringEventId, event.recurringEventId),
    ));
  if (owned.length > 0) continue; // skip — this recurring event is owned by a rolling job
}
```

One-off events and past already-imported rows are unaffected.

#### Promotion flow (user-initiated)

In `CalendarImportDialog`, every preview row whose event has a Google `recurringEventId` gets a third action next to "import / skip": **"ניהול כסדרה"** (track as rolling job).

Clicking it:
1. Fetch the parent recurring event from Google (using the stored `recurringEventId`) to read its RRULE.
2. Parse the RRULE into our `Cadence` shape via `parseGoogleRRule(rrule)`:
   - `FREQ=DAILY;INTERVAL=1` → `{ kind: "daily", interval: 1 }`
   - `FREQ=WEEKLY;INTERVAL=1;BYDAY=TU,TH` → `{ kind: "weekly", interval: 1, weekdays: [2, 4] }`
   - `FREQ=MONTHLY;INTERVAL=1;BYMONTHDAY=15` → `{ kind: "monthly", interval: 1, dayOfMonth: 15 }`
   - Anything else → returns `null`.
3. Open the rolling-job creation form pre-filled: title from event summary, description from event description, cadence from RRULE (or empty if unparseable, with a toast: *"לא הצלחנו לקרוא את דפוס החזרה — אנא בחר ידנית"*), start date from event `dtstart`, `sourceCalendarRecurringEventId` and `sourceCalendarId` set.
4. On save: normal `createRollingJobAction` path.

### Nudge integration

The smart-nudges engine in `apps/web/lib/nudges/` currently flags `draft`/`unpaid` rows. Update `compute.ts` to skip rows where `rollingJobId IS NOT NULL AND date > today`. A future unpaid rolling-job row isn't overdue — it hasn't happened yet. Once `date <= today`, nudges behave normally.

## UX

### Placement

- **Web:** a subtle button in the `IncomeHeader` row next to "Import from calendar" → opens `RollingJobsDialog` (list of active + paused jobs, create/edit, delete).
- **iOS:** an icon button (SF Symbol `arrow.triangle.2.circlepath`) in the Income tab's `GreenNavBar`, presenting a `RollingJobsView` sheet. Same view also linked from the Settings tab under "Data".

### Creation form (identical shape on both platforms, RTL)

```
Title                     [text]
Description (on entries)  [text]
Client (optional)         [client picker]
Category (optional)       [category picker]
Amount (₪)                [number]  Includes VAT? [toggle]  VAT rate [18]
Cadence                   [segmented: daily / weekly / monthly]
  ↳ daily:   every [N] days
  ↳ weekly:  every [N] weeks on [Sun Mon Tue Wed Thu Fri Sat]   (chip toggles)
  ↳ monthly: every [N] months on day [1..31]
Start date                [date picker, default = today]
End date                  [date picker, optional, clearable]
Notes                     [textarea, optional]
```

Below the form, a live preview line: *"הכנסה תיווצר אוטומטית: 14/4, 21/4, 28/4, 5/5 ..."* — first four upcoming occurrences rendered from the same `generateOccurrences` function used on the server. Kills ambiguity about what the cadence will actually do.

On save, the first 3 months of rows are materialized synchronously and appear in the income list immediately.

### Income list presentation

- Rolling-job rows get a subtle `Repeat` icon (lucide on web, `arrow.triangle.2.circlepath` on iOS) next to the description, muted color.
- Tapping/clicking the icon opens the **rolling job editor**, not the row editor — fast path to "change the rate going forward".
- Rows where `detachedFromTemplate = true` show the icon in an even more muted style. Hover tooltip (web) / long-press (iOS): *"נותק מתבנית הסדרה"*.
- A **"הסתר עתיד"** toggle is added to `IncomeFilters`. Default: OFF (future rows visible — the user wants to see them). State persisted in `localStorage` on web, `UserDefaults` on iOS. When ON, the list query gets a `WHERE date <= today` clause.

### Delete confirmation

Two-option dialog:
- *"שמור רשומות עבר"* (default) — keeps all past rows, detaches them.
- *"מחק גם רשומות עתידיות לא משולמות"* — also deletes future `draft` + `unpaid` rows.

## Edge cases (decided)

| Case | Behavior |
|---|---|
| Monthly + day 31 in short month | Clamp to last day of month (Feb 28/29, Apr 30) |
| End date before next occurrence | Generator returns `[]`; cron no-op; UI badge "הסתיימה" |
| Rolling job created with past `startDate` | Backfill all occurrences from `startDate` through horizon in one synchronous insert |
| High-frequency config | `generateOccurrences` returns at most 400 dates per call |
| Cron catches up after downtime | Idempotent via unique index; missed day inserts on the next run |
| User deletes a referenced client | `clientId` set to `NULL` on both the job and its future rows; snapshot `clientName` persists |
| Pause and resume months later | On resume, cron fills from `today` forward. No retroactive backfill |
| User changes `userSettings.timezone` | Existing rows unchanged (date column); future generations use new tz |
| Linked Google recurring event deleted in Google | Importer no longer sees it; rolling job keeps generating locally until paused/deleted |
| Unparseable RRULE on promotion | Form opens with cadence empty + toast; user picks manually; no hard failure |

## Error handling

- All inputs validated via Zod (`@seder/shared`). Server actions and API routes reuse existing `_lib/errors.ts` helpers.
- Cron failures handled per-job: `try/catch` → `Sentry.captureException` → continue. End-of-run summary logged.
- Materialization runs in a single DB transaction — partial writes impossible.
- Optimistic UI on the income list and rolling-jobs dialog follows the existing pattern in `IncomePageClient.tsx`.

## Testing

### Unit (Vitest, `@seder/shared`)

- `generateOccurrences`:
  - daily, interval 1 and 3
  - weekly, interval 1 and 2, every weekday combination (Mon-only, Sun–Thu, all 7)
  - monthly, interval 1 and 2, `dayOfMonth` in short months
  - `endDate` hard stop before horizon
  - `skipBefore` produces disjoint results from a prior run
  - 400-occurrence cap enforced
- `parseGoogleRRule` — three supported shapes + one unsupported returning `null`.

### Integration (Vitest, `apps/web`)

- `materializeRollingJob` against a test DB:
  - First run inserts N rows, second run inserts 0 (idempotency).
  - Pause + resume doesn't backfill.
  - Template field edit rewrites only attached future rows.
  - Detached rows survive subsequent template edits.
- `updateRollingJobAction`:
  - Cadence change reconciles attached future rows correctly.
  - Field-only change issues a single UPDATE.
- `createRollingJobAction` with a past `startDate` backfills as expected.
- Calendar importer skips events with matching `sourceCalendarRecurringEventId`.
- Nudge engine filters future rolling-job rows from overdue calculations.
